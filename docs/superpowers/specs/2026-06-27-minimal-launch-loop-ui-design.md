# UsageMeter 最小上线闭环 UI 设计

日期：2026-06-27

## 该不该做

该做，但只做最小上线闭环 UI，不扩展成完整 SaaS 后台。

当前后台已经有仪表盘、租户、套餐、API Key、用量、收益、账单、接入文档、日志和设置。继续补普通 CRUD 页收益不高。上线前真正缺的是运营闭环：账单如何生成、额度风险如何发现、异常如何处理。

## 为什么

README 中明确下一阶段是 `X-Request-Id` 幂等计费、每日额度限流、`UsageDailyAggregate` 原子更新、账单幂等生成。现有前端能展示数据，但缺少围绕这些能力的操作入口和状态视图。

本设计补三页：

- 账单生成中心：承接账单幂等生成和重跑入口。
- 额度与限流监控：承接每日额度限流和用量风险观察。
- 异常处理中心：把请求、鉴权、限流、账单异常从日志里聚合出来，便于运营排查。

## 怎么做

### 导航结构

在现有后台侧边栏新增三个一级入口：

- `额度监控`：放在 `用量` 后面。
- `账单生成`：放在 `账单` 后面。
- `异常中心`：放在 `日志` 前面或作为 `日志` 前的独立入口。

保留现有 `日志` 页面用于明细检索。新增页面负责聚合、风险和操作入口，不替代日志。

### 页面一：账单生成中心

路由建议：`/billing-runs`

目标：让运营按账期查看哪些租户需要生成账单，并在后端能力完成后触发生成或重跑。

页面结构：

- 顶部筛选：账期、租户、生成状态。
- 指标条：待生成、草稿、已出账、失败。
- 主表格：租户、账期、用量、超额用量、预计金额、当前账单状态、最后生成时间、操作。
- 操作区：生成账单、查看账单、重跑失败。

本轮 UI 边界：

- 如果后端没有 `POST /tenants/:tenantId/invoices/generate`，生成和重跑按钮显示禁用态，并提示“账单生成接口待实现”。
- 列表数据先复用 `listTenants`、`listDailyUsage`、`listInvoices` 推导。
- 已存在账单跳转到 `/invoices/:id`。

### 页面二：额度与限流监控

路由建议：`/limits`

目标：让运营快速发现接近每日额度或已经超限的租户。

页面结构：

- 顶部指标：今日请求数、今日消耗额度、风险租户数、已触发限流次数。
- 筛选：日期、租户、风险等级。
- 租户额度表：租户、套餐、每日额度、今日已用、剩余额度、使用率、风险等级、最近请求时间。
- 右侧或下方趋势：按日展示请求数和额度消耗。

风险等级规则：

- 正常：使用率 `< 80%`。
- 预警：使用率 `>= 80%` 且 `< 100%`。
- 超限：使用率 `>= 100%`。

本轮 UI 边界：

- 使用 `listDailyUsage` 和 `listTenants` 中的 `plan.dailyUnitLimit` 计算使用率。
- 429 事件从 `listUsageEvents` 的 `statusCode === 429` 推导；如果当前数据没有 429，显示空状态。
- 不实现真实限流开关和策略编辑。

### 页面三：异常处理中心

路由建议：`/exceptions`

目标：把散落在请求日志、账单状态、API Key 状态中的异常归并为可处理列表。

页面结构：

- 顶部分类指标：重复请求、鉴权失败、撤销 Key 调用、限流触发、账单失败。
- 筛选：异常类型、租户、严重级别、日期。
- 异常表格：级别、类型、租户、关联对象、摘要、发生时间、处理状态、操作。
- 操作：跳转请求日志、跳转账单详情、跳转租户详情。

异常推导规则：

- 鉴权失败：`UsageEvent.statusCode` 为 `401` 或 `403`。
- 限流触发：`UsageEvent.statusCode === 429`。
- 服务异常：`UsageEvent.statusCode >= 500`。
- 撤销 Key 调用：请求事件关联的 API Key 状态为 `REVOKED`。
- 账单异常：账单状态为失败类状态时展示；当前模型没有失败状态，本轮不伪造失败，只保留空状态和后续扩展位。
- 重复请求：依赖后端幂等实现后的重复标记；当前唯一约束已存在，本轮只保留空状态说明。

本轮 UI 边界：

- 不新增异常数据库表。
- 不实现处理状态持久化。
- 不新增通知、Webhook、工单流。

## 数据来源

- `listTenants()`：租户、套餐、每日额度。
- `listDailyUsage()`：每日请求数和额度消耗。
- `listUsageEvents()`：请求状态、requestId、接口、API Key 前缀。
- `listInvoices()`：账单状态、金额、账期。

当前阶段优先复用前端聚合。后端分页、过滤和专用聚合接口可在数据量增长后再补。

## UI 风格

沿用现有后台风格：

- 左侧固定导航。
- 表格为主，指标卡为辅。
- 状态使用现有 `status-pill` 类体系。
- 操作按钮使用 lucide 图标和现有按钮样式。
- 页面保持运营工具密度，不做营销式 hero 或装饰背景。

## 测试

前端测试补充：

- 新导航入口可见。
- 三个新增页面能在 mock API 数据下渲染。
- 账单生成按钮在接口未实现时为禁用或提示态。
- 额度监控能正确显示正常、预警、超限状态。
- 异常中心能从 401、403、429、500 请求中推导异常类型。

## 不做

- 不接真实支付。
- 不做后台任务系统。
- 不做通知、Webhook、邮件。
- 不做租户成员、角色矩阵和客户门户。
- 不新增后端业务算法。
